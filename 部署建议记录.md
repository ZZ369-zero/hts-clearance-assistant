# 部署建议记录

## 当前结论

本项目目前是 Node 后台 + 前端页面结构，包含接口、税率查询、数据同步中心、232/AD-CVD/棉费等数据抓取逻辑。因此：

- GitHub Pages 只能托管静态页面，不能长期运行 `server.js` 后台。
- 如果需要电脑关机后仍能被外部访问，推荐使用云端运行环境，例如 Railway、Render、Fly.io 或 VPS。
- GitHub 可以作为代码仓库和自动部署触发源，但不能单独替代后台服务器。

## 推荐部署优先级

1. Railway
   - 适合当前项目结构，改动少。
   - 可直接从 GitHub 自动部署。
   - 适合保留后台接口和数据同步中心。

2. Render
   - 同样适合 Node Web Service。
   - 部署流程简单，适合快速上线测试。

3. Fly.io
   - 更接近正式云服务部署。
   - 可控性更强，但配置复杂度略高。

4. VPS 云服务器
   - 适合长期正式使用或商用。
   - 需要自行维护 Node、PM2、防火墙、日志和备份。

5. GitHub Pages + GitHub Actions
   - 成本低，但不是实时后台。
   - 需要改造成“静态页面 + 定时生成 JSON 数据”的模式。
   - 适合接受 1 小时到 24 小时级别数据延迟的场景。

## 已采纳的数据刷新频率

- USITC HTS / 普通税率 / Chapter 99：每 1 小时。
- 232 Metals HTS List：每 6 小时。
- AD/CVD 官方站点监控：每 24 小时，避免过于频繁访问官方站点。
- 棉费 Import Assessment：每天 1 次。
- 中文翻译缓存 / 商品描述辅助：每周或按新增数据触发。

## GitHub Pages + GitHub Actions 方案说明

GitHub Pages 不会因为仓库 60 天没有更新而停止网页访问。真正可能停止的是 GitHub Actions 的定时任务 `schedule`。

如果定时任务被停用：

- 网站页面仍可访问。
- 已生成的 HTML/CSS/JS/JSON 仍可读取。
- 自动抓取和自动刷新数据会停止。
- 用户看到的数据会停留在最后一次成功生成的版本。

因此，该方案的主要风险不是“网站打不开”，而是“数据过期”。

## GitHub Actions 触发建议

不建议把定时任务设置在每小时整点，例如 `0 * * * *`，因为 GitHub Actions 整点附近更容易排队或延迟。

如果使用每 1 小时刷新，可以错峰执行：

```yaml
on:
  schedule:
    - cron: "7 * * * *"
  workflow_dispatch:
```

如果需要多类数据不同频率，建议拆成多个 workflow：

```yaml
# HTS / 普通税率 / Chapter 99：每 1 小时
- cron: "7 * * * *"

# 232 Metals HTS List：每 6 小时
- cron: "17 */6 * * *"

# AD/CVD 官方站点监控：每天 1 次
- cron: "27 2 * * *"

# 棉费 Import Assessment：每天 1 次
- cron: "37 3 * * *"

# 中文翻译缓存：每周 1 次
- cron: "47 4 * * 1"
```

以上 cron 默认按 UTC 时间运行。部署时需要按中国时间换算，或在脚本内部处理时区。

## 避免 60 天无活动停用的建议

如果采用 GitHub Pages + GitHub Actions，建议设计为：

```text
定时抓数据 -> 生成 data/*.json -> 自动 commit 回仓库 -> GitHub Pages 读取这些 JSON
```

这样每次数据更新都会产生仓库活动，可以降低定时任务因长期无活动被停用的风险。

如果只使用 Pages artifact 部署，不产生 commit，则更容易出现仓库长期无活动后定时任务被停用的情况。

## 最终建议

如果目标是“真实在线查询 + 后台自动更新 + 同事稳定访问”，优先选择 Railway / Render / Fly.io / VPS。

如果目标是“低成本展示 + 可接受非实时数据”，可以选择 GitHub Pages + GitHub Actions，并把数据更新结果自动提交到仓库。
